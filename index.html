CasTL_Reading

发表于 2024-12-09 分类于 -论文阅读 -LLM&TAMP

引言

今天详细地读了一下这篇文章CasTL，主要的内容是提出了一个基于IMTMP这一TAMP规划器的LLM辅助规划器，主要应用在有长效任务需求的场景中。这篇文章的核心点跟我现在所做的内容有些像，具体在于：用LLM提取任务的约束，并在后续利用约束来完成任务（这一能力主要来自于IDTMP这款基于约束的规划器）。所以我看看能否借鉴一些地方。

论文主要内容

动机

  1. 现如今的基于LLM的TAMP规划器一般都是让LLM直接输出动作序列。受限于LLM自身的规划（主要是逻辑推理和数学计算）能力，其很难解决存在复杂，非单调逻辑的有长效要求的TAMP问题。（的确，这个动机与我现在的项目几乎一样）
  2. 在实际的任务当中，用户往往会加入一些额外的约束（比如指定哪个物体不能动，哪些区域不能去，个人理解这其实是一个非常大的领域，如果展开做内容相当多），如何兼顾这些实际存在的约束和完成任务，现在很少有人提及到，尤其是在使用LLM辅助机器人规划这一领域上。

做法概述

这个规划框架的核心是2018年Lydia E. Kavraki实验室提出的基于约束的TAMP规划器IDTMP。可以说，这篇文章的主要内容就是：如何把一个包含自然语言目标和自然语言约束的长效规划问题，转换成IDTMP这一规划器所能解决的TAMP问题。这也是我想着重参考的地方，至于IDTMP规划器的内在工作原理则作为次要的参考。

做法的关键点

对约束的分类

  1. Attribute Constraints: 源于物体属性的约束。以一个例子解释，带有床的卧室。表示“卧室”这一物体（object）的约束为带有床的（with bed）。个人对这个约束的定义比较疑惑，或许是为了符合IDTMP的规划过程。
  2. Eventual Constraints: 相当于任务目标。在规划完成后必须要满足的约束，或者说，不管完成任务的过程中真值是否为1,结束时必须为1。
  3. Global Constraints: 在执行过程中必须满足的约束，和2相对应，完成任务的过程中和结束时必须都为1.
  4. Implication Constraints: 用户提出的特定的约束，可以是执行动作的先后顺序。

其实，从PDDL的三大元素（_object, predicate, action_）可以大概理解出来作者对这四类约束的定义动机：首先每一个约束都是由谓词（_predicate_）和个体（_object_）组成的命题;其次，第三类约束可以理解为向每一个_action_中的_pre_加入了该约束对应的命题；最后，第四类约束是对特定几个_action_的_pre_和_eff_进行修改，使其满足一定的顺序要求。

将自然语言的约束转化为机器语言

  1. 左侧的LLM for PDDL Generation，与LLM+P比较相似：将自然语言的Problem（也就是任务目标）转化为PDDL格式的Problem.pddl文件，当然，domain.pddl文件都是提前设定好的。原文中还提到与LLM+P的差别：

The key difference between CaStL and LLM+P in this module is that we use the output of the prior multi-step query to disambiguate the problem as input.

说实话目前没理解到这个区别具体是啥。

  1. 右侧的LLM for Script Generation，本质上是通过一个LLM实现将自然语言转化为PDDL语言，但是原文提到由于LLM的训练文本很少包含PDDL3.0版本之后的数据，因此直接转化的效果不好。因而通过LLM生成代码，利用自己建立的一个功能包实现这一功能。原文如下：

Thus, the success rate for LLMs to directly translate constraints into PDDL (i.e., the :constraints field) is low. Therefore, we instead prompt the LLM to extract and translate constraints into a Python script that uses a custom API to build logical expressions and add constraints to the task planner.

个人总结

相比于自己的项目，本文可借鉴的地方主要在动机+约束分类上。这四种约束虽然不见得都可以借鉴到自己的项目中，但是一定程度上开拓了思路：现在我做到的只是将目标状态进行拆解从而形成子任务引导LLMTAMP规划器，是不是可以像这篇文章一样在目标状态之外加入全局约束(Global constraints)和人为特定约束(Implication constraints)，再设计一个LLM参与的约束管理层将这三者分开处理，从而实现更高级的子任务分配？

P.S.个人并不认为通过本地的以PDDL为基础的TAMP规划器完成任务是比较make sense的做法，因为本质上是解耦的LLM翻译器+本地现成的TAMP规划器，贡献点很容易理解成：如何通过LLM将一个自然语言TAMP问题翻译成现有的TAMP规划器可以解决的问题，从而被质疑缺乏创新。
